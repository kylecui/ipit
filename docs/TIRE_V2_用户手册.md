# TIRE V2 用户手册

## 1. 文档说明

### 1.1 文档用途

本文档用于说明 **TIRE V2（Threat Intelligence Reasoning Engine V2，威胁情报推理引擎 V2）** 的安装部署方式、系统功能、角色权限、操作流程、配置方法、报表输出及日常使用注意事项。

本文档既可作为：

1. 软件产品用户使用说明书；
2. 计算机软件著作权登记时的软件鉴别材料之一。

### 1.2 适用版本

- 软件名称：TIRE V2
- 文档语言：中文
- 适用对象：系统管理员、普通用户、运维人员、测试人员、软著登记材料整理人员

### 1.3 系统简介

TIRE V2 是一套面向 IP 威胁情报分析场景的软件系统，支持对 IP 地址进行多源威胁情报查询、语义识别、证据融合、风险评分、历史留存、报告生成、权限管理及 API 用量统计。

系统采用插件化架构，通过多个情报插件采集外部与内部数据，经标准化、语义增强、分析器评分与证据融合后，形成统一的威胁判定结果，并支持生成 HTML 详细报告与历史对比结果。

---

## 2. 系统目标与应用场景

### 2.1 系统目标

本系统主要用于：

- 对目标 IP 地址进行威胁情报聚合查询；
- 对查询结果进行统一分析和风险判定；
- 生成带证据链的分析结果与详细报告；
- 为管理员提供插件、用户、密钥、LLM、日志、统计信息等管理能力；
- 为普通用户提供自主分析、查看历史、生成报告、配置个人密钥等能力。

### 2.2 典型应用场景

- 安全运营中心（SOC）进行 IP 风险初判；
- 威胁分析人员对可疑 IP 进行情报补充与归因分析；
- 测试/研发团队验证多源情报整合逻辑；
- 管理员统一管理共享 API Key、共享 LLM、用户权限与用量情况；
- 作为内部威胁情报辅助分析平台进行日常使用。

## 4. 系统架构概述

### 4.1 整体架构

TIRE V2 的整体处理流程如下：

```text
用户输入（Web / API / CLI / 批量）
        ↓
QueryEngine 查询编排
        ↓
插件采集（Plugins）
        ↓
数据标准化 / 语义增强
        ↓
分析器（Analyzers）评分与证据融合
        ↓
Verdict 判定结果
        ↓
报告输出（HTML / JSON / Markdown / CLI / Narrative Report）
```

### 4.2 主要组成部分

系统主要由以下模块组成：

1. **应用核心模块（app/）**
   - 提供 Web 接口、API 接口、配置管理、查询编排、LLM 调用等能力。

2. **插件模块（plugins/）**
   - 负责从不同威胁情报源采集原始数据。

3. **模型模块（models/）**
   - 提供 Pydantic 数据模型与统一数据契约。

4. **分析模块（analyzers/）**
   - 负责信誉评分、上下文分析、噪声识别、证据融合与最终判定。

5. **报告模块（reporters/）**
   - 负责将分析结果输出为多种格式。

6. **模板与国际化模块（templates/、locales/）**
   - 提供 Web 页面与中英文界面文本。

7. **管理后台模块（admin/）**
   - 提供用户、策略组、插件、API Key、LLM、日志、统计等后台能力。

8. **持久化模块（storage/）**
   - 负责查询快照、报告、API Key、用量统计等持久化存储。

## 5. 系统安装与部署

### 5.1 本地开发部署

#### 步骤 1：获取代码

```bash
git clone <repository-url>
cd tire
```

#### 步骤 2：创建虚拟环境并安装依赖

```bash
uv venv
# Linux/macOS
source .venv/bin/activate
# Windows
# .venv\Scripts\activate

uv pip install -r requirements.txt
```

#### 步骤 3：准备配置文件

```bash
cp .env.example .env
```

根据实际需要修改 `.env` 中的基础配置，例如：

- `ADMIN_PASSWORD`
- `SESSION_SECRET_KEY`
- `TIRE_PORT`
- `ROOT_PATH`
- `LANGUAGE`
- `TIRE_FERNET_KEY`
- 默认 LLM 环境回退参数（如启用）

> 说明：V2 中插件 API Key 与 LLM Key 的主要配置方式为数据库配置；`.env` 中相关项仅作为兼容性回退来源或系统基础配置来源。

#### 步骤 4：启动服务

```bash
uv run uvicorn app.api:app --reload
```

浏览器访问：

```text
http://127.0.0.1:8000/
```

### 5.2 Docker 部署

#### 步骤 1：准备配置

```bash
cp .env.example .env
vim .env
```

#### 步骤 2：启动容器

```bash
docker compose up -d
```

#### 步骤 3：验证服务状态

```bash
curl http://127.0.0.1:8000/healthz
```

预期返回：

```json
{"status":"healthy","service":"threat-intel-reasoning-engine"}
```

### 5.3 V1 / V2 并行部署说明

当前项目要求支持 V1 与 V2 并行运行：

- V1 运行在根路径 `/`
- V2 运行在子路径 `/v2`

部署 V2 时应确保：

1. `ROOT_PATH=/v2`
2. 反向代理已将 `/v2` 正确转发到 V2 应用
3. 不得影响 V1 正常访问与使用

### 5.4 常用运维命令

```bash
# 启动/更新
docker compose up -d --build

# 查看状态
docker compose ps

# 查看日志
docker compose logs -f tirev2-app

# 停止服务
docker compose down
```

## 7. 系统首页与分析功能

### 7.1 首页功能

系统首页为分析主界面，主要包含：

- IP 输入框；
- 刷新缓存选项；
- 分析按钮；
- 最近查询列表；
- 最近报告列表；
- 历史对比入口。

### 7.2 发起分析

操作步骤：

1. 打开系统首页；
2. 在输入框中输入目标 IP 地址；
3. 如需绕过缓存，勾选“刷新缓存”；
4. 点击“分析 IP”；
5. 系统开始调用已启用插件进行数据采集与分析。

### 7.3 分析结果说明

分析完成后，页面会展示：

- 分析对象基本信息；
- 最终评分；
- 判定等级；
- 置信度；
- 推荐处置动作；
- 证据详情；
- 已查询数据源状态；
- 归属信息与上下文信息。

### 7.4 评分等级说明

系统采用 0–100 分制：

| 分值区间 | 等级 | 说明 |
|---|---|---|
| 0–20 | 低 | 风险较低 |
| 21–45 | 中 | 存在一定风险 |
| 46–75 | 高 | 风险较高 |
| 76–100 | 严重 | 高危，需要立即关注 |

## 8. 详细报告功能

### 8.1 报告生成

用户在完成一次分析后，可点击“生成详细报告”。

系统会基于**已完成的分析结果**生成报告，不会在报告生成阶段重新发起新的情报查询。

### 8.2 报告内容

详细报告通常包含以下内容：

- 生成时间；
- 查询时间；
- 评分与判定摘要；
- 基础归属分析；
- 域名关联分析；
- 威胁情报观察；
- 风险评估；
- 处置建议；
- 结论与参考；
- AIGC 免责声明（如启用 LLM）。

### 8.3 LLM 增强与模板回退

详细报告支持两种模式：

1. **LLM 增强报告**
   - 在已配置可用 LLM 且当前用户具备使用权限时启用。

2. **模板报告**
   - 当未配置 LLM、LLM 超时、权限不允许或上游失败时，自动回退到模板报告。

### 8.4 报告缓存与历史

系统会保存已生成报告，并支持历史查看与比较。

相同用户、相同数据快照、相同语言、相同 LLM 指纹下的报告可直接复用；如用户选择重新生成，则会归档旧报告并生成新版本。

## 9. 历史查询与对比功能

### 9.1 最近查询

首页会展示当前用户最近的查询记录，包括：

- IP 地址；
- 最终评分；
- 判定等级；
- 查询时间；
- 查看与对比操作。

### 9.2 最近报告

首页会展示最近生成的报告记录，包括：

- 目标 IP；
- 报告类型；
- 生成时间；
- 查看与对比操作。

### 9.3 时间线与对比

系统支持以下历史对比能力：

1. **时间线视图**
   - 查看同一 IP 多次查询后的评分变化。

2. **快照对比**
   - 对比两个不同分析快照之间的差异。

3. **报告对比**
   - 对比两个历史报告的内容变化。

## 12. API Key 与 LLM 配置说明

### 12.1 插件 API Key 配置原则

V2 中插件 API Key 采用数据库配置模式，支持：

- 用户个人 Key；
- 管理员共享 Key；
- 权限继承控制。

系统的插件 Key 解析优先级为：

```text
用户个人 Key → 允许使用的共享管理员 Key → 无
```

在“仅使用已配置密钥”模式下，不再回退到环境变量。

### 12.2 LLM 配置原则

LLM 支持以下来源：

- 个人 LLM
- 用户指定共享 LLM
- 组分配共享 LLM
- 环境变量回退（如未关闭）
- 模板模式

### 12.3 数据加密

数据库中的插件 Key 与 LLM Key 为加密存储，解密依赖系统运行时的 Fernet 配置。

## 17. 故障排查

### 17.1 无法访问首页

排查方向：

- 检查应用是否已启动；
- 检查 `healthz` 是否正常；
- 检查反向代理与端口映射；
- 检查 `ROOT_PATH` 配置是否正确。

### 17.2 登录后跳转异常

排查方向：

- 检查 `ROOT_PATH` 是否与部署路径一致；
- 检查代理层是否保留 `/v2` 前缀；
- 检查 session 是否正常写入。

### 17.3 某插件没有结果

排查方向：

- 检查插件是否启用；
- 检查是否配置了 API Key；
- 检查 API 额度是否耗尽；
- 检查后台日志中的错误信息；
- 检查数据源状态区域是否显示失败原因。

### 17.4 LLM 报告未生成

排查方向：

- 检查 LLM Key、Base URL、Model 是否正确；
- 检查用户权限是否允许使用 LLM；
- 检查上游接口是否超时或失败；
- 检查是否自动回退为模板报告。

### 17.5 API 用量统计无数据

排查方向：

- 确认已发生真实插件或 LLM 调用；
- 确认查看账户是否具备访问对应统计的权限；
- 检查后台 `/admin/api/usage` 返回结果；
- 检查 `results.db` 中事件表是否存在记录。
