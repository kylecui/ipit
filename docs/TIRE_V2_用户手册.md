## 2. 软件概述

TIRE V2 是一套用于 IP 威胁情报分析的软件系统。系统通过插件方式接入多个威胁情报源，对指定 IP 地址进行多源查询，并将采集到的数据进行统一标准化、语义增强、分析评分与证据融合，最终输出统一的风险等级和分析结果。

与传统“仅查询、仅展示”型工具不同，TIRE V2 不仅能够收集情报结果，还能够：

- 对不同来源的结果进行统一结构化；
- 将情报结果转化为可追溯的证据项；
- 基于证据进行风险评分与等级判定；
- 生成详细 HTML 报告与历史对比结果；
- 提供后台管理、用户权限控制、Key 管理与用量统计。

系统提供 Web 页面、REST API、CLI 等多种访问方式，其中 Web 页面和管理后台是当前最主要的使用方式。

## 3. 软件目标与适用场景

### 3.1 软件目标

本软件旨在实现以下目标：

1. 将多个威胁情报源统一纳入一个可管理、可扩展的分析平台；
2. 对目标 IP 提供可解释的分析结果，而不是简单原始查询结果堆叠；
3. 为管理员提供可视化的插件配置、用户权限管理、LLM 配置与 API 用量统计能力；
4. 为普通用户提供分析、报告、历史对比与个人配置能力；
5. 支持 V1 与 V2 并行部署，确保新版本演进不影响旧版本正常使用。

### 3.2 适用场景

本软件适用于如下业务场景：

- 威胁情报研判；
- 安全运营中心（SOC）风险初筛；
- 网络安全事件分析与归因辅助；
- 安全产品验证与多源情报聚合测试；
- 内部分析平台建设与管理。

## 5. 安装与部署说明

### 5.1 本地开发部署

#### 步骤 1：获取源码

```bash
git clone <repository-url>
cd tire
```

#### 步骤 2：创建虚拟环境

```bash
uv venv
source .venv/bin/activate
# Windows 下可使用 .venv\Scripts\activate
```

#### 步骤 3：安装依赖

```bash
uv pip install -r requirements.txt
```

#### 步骤 4：准备配置文件

```bash
cp .env.example .env
```

根据部署需要编辑 `.env`，重点关注：

- `ADMIN_PASSWORD`
- `SESSION_SECRET_KEY`
- `ROOT_PATH`
- `TIRE_PORT`
- `LANGUAGE`
- `TIRE_FERNET_KEY`

#### 步骤 5：启动服务

```bash
uv run uvicorn app.api:app --reload
```

### 5.2 Docker 部署

#### 步骤 1：准备环境变量

```bash
cp .env.example .env
vim .env
```

#### 步骤 2：启动服务

```bash
docker compose up -d
```

#### 步骤 3：健康检查

```bash
curl http://127.0.0.1:8000/healthz
```

预期返回：

```json
{"status":"healthy","service":"threat-intel-reasoning-engine"}
```

### 5.3 V1 / V2 并行部署要求

根据项目要求：

- V1 运行于根路径 `/`
- V2 运行于子路径 `/v2`

因此部署 V2 时应满足：

1. 设置 `ROOT_PATH=/v2`
2. 代理层将 `/v2` 请求转发给 V2 应用
3. 不能影响 V1 正常运行

### 5.4 常用运维命令

```bash
docker compose up -d --build
docker compose ps
docker compose logs -f tirev2-app
docker compose down
```

## 7. 系统主要功能说明

### 7.1 IP 威胁分析

用户输入目标 IP 后，系统自动调用已启用插件查询相关数据，结合信誉分析、上下文分析和噪声识别逻辑，输出统一的分析结论。

### 7.2 多源情报聚合

系统可同时接入多个情报源，对相同目标进行交叉验证，从而提升结果完整性和可解释性。

### 7.3 证据驱动判定

系统并非仅返回“高/中/低”结论，而是为每个判定输出详细证据项，包括：

- 证据来源；
- 证据类别；
- 严重等级；
- 详细说明；
- 分值影响。

### 7.4 详细报告生成

系统支持根据当前分析结果生成详细 HTML 报告，并支持在 LLM 可用时生成增强叙述版本。

### 7.5 历史留存与对比

系统会持久化存储查询快照与生成报告，并支持时间线、快照对比与报告对比。

### 7.6 插件配置与后台管理

管理员可通过后台管理插件启停、共享 Key、共享 LLM、用户权限和策略组。

### 7.7 API 用量统计

系统可记录插件 API 与 LLM 的调用次数、成功率、平均延迟、Token 使用量等信息，并按管理员/普通用户不同角色进行展示。

## 10. API 与访问路径说明

### 10.1 路径说明原则

TIRE V2 当前采用 **“内部路由保持原结构，外部通过 `ROOT_PATH=/v2` 子路径发布”** 的方式部署。

因此需要区分两个概念：

1. **代码内部定义路径**  
   例如：`/api/v1/ip/{ip}`

2. **V2 对外实际访问路径**  
   例如：`/v2/api/v1/ip/{ip}`

### 10.2 V2 对外实际访问路径

在 V2 正式部署场景中，应使用以下路径格式：

- `GET /v2/healthz`
- `GET /v2/readyz`
- `GET /v2/api/v1/ip/{ip}`
- `POST /v2/api/v1/analyze/ip`
- `GET /v2/api/v1/debug/sources/{ip}`
- `POST /v2/api/v1/report/generate`
- `POST /v2/analyze`
- `GET /v2/admin/...`

### 10.3 V1 / V2 路径对照表

| 功能 | V1 / 默认根路径 | V2 对外路径 |
|---|---|---|
| 首页 | `/` | `/v2/` |
| 分析提交 | `/analyze` | `/v2/analyze` |
| IP 查询 API | `/api/v1/ip/{ip}` | `/v2/api/v1/ip/{ip}` |
| 报告生成 API | `/api/v1/report/generate` | `/v2/api/v1/report/generate` |
| 后台登录 | `/admin/login` | `/v2/admin/login` |
| 健康检查 | `/healthz` | `/v2/healthz` |

### 10.4 接口用途说明

#### 健康检查接口

- `GET /v2/healthz`
- `GET /v2/readyz`

#### 分析接口

- `GET /v2/api/v1/ip/{ip}`
- `POST /v2/api/v1/analyze/ip`
- `POST /v2/analyze`

#### 报告接口

- `POST /v2/api/v1/report/generate`

#### 调试接口

- `GET /v2/api/v1/debug/sources/{ip}`

#### 用量统计接口

- `GET /v2/admin/api/usage`

管理员访问该接口时，可获取平台级统计；普通用户访问时，获取与自己相关的范围化统计。

## 13. LLM 与报告功能说明

### 13.1 报告生成原则

报告生成不会重新触发新的情报查询，而是使用已有分析结果与已保存快照进行渲染。

### 13.2 LLM 模式

系统支持：

- 个人 LLM
- 共享 LLM
- 模板回退

### 13.3 模板回退说明

当 LLM 不可用、权限不允许、超时或上游失败时，系统会自动回退到模板报告模式，以确保报告功能始终可用。

### 13.4 报告内容组成

报告内容通常包括：

- 评分与结论；
- 查询时间与报告生成时间；
- 基础归属分析；
- 证据摘要；
- 风险评估；
- 处置建议；
- AIGC 提示。

## 16. 数据库与存储说明

### 16.1 持久化目标

系统会持久化以下数据：

- 查询快照；
- 历史报告；
- 插件 API Key；
- 用户与权限数据；
- API 用量事件；
- LLM 用量事件。

### 16.2 主要数据库/存储位置

典型数据文件包括：

- `storage/results.db`
- `cache/cache.db`
- `admin/admin.db` 或容器数据卷映射目录

### 16.3 数据内容分类

#### 查询快照

用于记录一次分析的最终结果和原始来源数据。

#### 历史报告

用于保存生成过的 HTML 报告及其关联快照信息。

#### Key 配置数据

用于保存共享与个人的插件 Key / LLM Key，加密存储。

#### 用量事件数据

用于记录插件与 LLM 的调用明细，为统计页面与接口提供数据来源。

### 16.4 数据安全说明

API Key 与 LLM Key 使用加密方式存储。运行时依赖稳定的 Fernet 配置进行解密。

## 17. 日常运维与故障排查

### 17.1 应用无法启动

检查：

- `.env` 配置是否正确；
- Docker 容器是否正常运行；
- 端口是否被占用；
- 依赖是否已正确安装。

### 17.2 V2 路径访问异常

检查：

- `ROOT_PATH=/v2` 是否设置；
- 代理是否保留 `/v2` 路径；
- 登录重定向是否保持 `/v2` 前缀。

### 17.3 某插件无结果

检查：

- 插件是否启用；
- 是否配置共享或个人 Key；
- 上游额度是否耗尽；
- 数据源状态表是否显示错误原因。

### 17.4 报告未启用 LLM

检查：

- LLM 权限是否允许；
- LLM Key / URL / Model 是否有效；
- 是否自动回退为模板模式。

### 17.5 用量统计无记录

检查：

- 是否发生了真实插件或 LLM 调用；
- 当前账号是否有权查看该范围数据；
- 后台 JSON 统计接口是否有数据；
- 事件表是否已有记录。

## 18. 名词解释

### 18.1 Threat Intelligence（威胁情报）

指关于恶意 IP、域名、攻击基础设施、恶意行为等可用于安全分析的数据与知识。

### 18.2 Observable（观测对象）

本系统当前主要指用户输入的 IP 地址，也可扩展为其他类型的安全观测对象。

### 18.3 Plugin（插件）

系统中接入不同情报源的功能模块，用于查询外部或内部数据。

### 18.4 Verdict（判定结果）

系统基于证据和评分逻辑输出的统一分析结论。

### 18.5 Evidence（证据项）

每条判定依据的结构化表示，包含来源、说明、严重程度、分值影响等内容。

### 18.6 Snapshot（快照）

系统对一次分析结果的持久化保存记录，用于历史比对与报告生成。

### 18.7 Narrative Report（叙述性报告）

基于分析结果生成的详细 HTML 报告，可由模板直接生成，也可通过 LLM 增强生成。

### 18.8 Shared Key（共享密钥）

管理员统一配置、允许普通用户按策略使用的 API Key。

### 18.9 Personal Key（个人密钥）

普通用户为个人账户单独配置的 API Key。

### 18.10 LLM

大语言模型，用于生成增强型叙述报告内容。

### 18.11 Root Path

应用部署时的外部前缀路径。在当前项目中，V2 通过 `ROOT_PATH=/v2` 发布到 `/v2` 子路径下。
